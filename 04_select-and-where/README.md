# S-04: `select()` and `where()` — SQLAlchemy 2.0 Style

> **Unit goal**: Master the 2.0-style `select()` / `where()` API, understand
> how the ORM translates Python expressions into SQL, and learn to avoid the
> N+1 problem with `selectinload()`.
>
> **学習目標**：2.0スタイルの `select()` / `where()` を習得し、ORMがPython式をSQLに変換する仕組みを理解する。
> `selectinload()` によるN+1問題の回避も実践する。

## Topics covered / 学習トピック

| Topic | SQLAlchemy API |
|---|---|
| Basic SELECT | `select(Model)` |
| Filtering | `.where(Model.col == value)` |
| AND / OR / NOT | `and_()`, `or_()`, `not_()` |
| Ordering | `.order_by(Model.col.desc())` |
| Limit / Offset | `.limit(n).offset(m)` |
| Scalar result | `session.scalar()`, `session.scalars()` |
| Eager loading | `selectinload(Model.rel)` |
| Aggregation | `func.count()`, `func.sum()` |
| 1.x → 2.0 | `Query` API → `select()` |

## Key Interview Points / 面接頻出ポイント

- 💡 `session.execute(select(M))` vs deprecated `session.query(M)` — always use `select()` in 2.0
- 🔍 N+1 problem: loading `relationship` inside a loop without eager-load causes 1+N SQL queries
- 💡 `scalars().all()` vs `execute().all()` — difference in return type (ORM object vs Row)
- 💡 `expire_on_expire=False` prevents lazy-load errors after `commit()` in async context
